- Customer Service App Development: Christopher Baron and Shiovani have identified extra features needed for the customer service application, which Michael Weaver is currently working on for the offshore Philippine customer support team. Christopher Baron will take the current ticket and refine it into a task for Michael Weaver, which will include both front-end and back-end aspects to better align on the customer support application's direction. This application is being developed to handle overnight calls.
    
- Smoke Detector Alert Testing: Christopher Baron intends to start a fire this week to test the smoke detector alert functionality, as they are extra paranoid, even though the code appears fine. They anticipate that this item, related to a little test smoke detector showing up as an alert, will be closed out this week.
    
- Caseload Editing and Profile Demo Experience: Regarding caseload editing, Christopher Baron reviewed Shiovani's feedback, added one comment, and is fine with making a separate pull request (PR). For the profile demo experience, Christopher Baron looked at the front-end code and found it reasonable, suggesting they should merge what Michael Weaver has and then follow up on data seeding. They discussed whether to use real data or fake data for the demo, with Michael Weaver leaning towards making the system work with real data.
    
- Data Strategy for Profile Demo: The group debated the best approach for the profile demo data, with Michael Weaver preferring to make the system function as if using "real" data to avoid maintaining separate data structures on the front end. Christopher Baron initially opposed using fake data in the database but also wanted to avoid clearing out existing test agency data upon switching to a demo mode. They agreed to strategize again on the data implementation, noting the need to have this feature ready for "Progress" so customers do not see the old caseload experience.
    
- Provisioning App Phase One and Internal Workflows: Christopher Baron informed Michael Weaver about the provisioning app, which Ritwik Rastogi is working on, detailing the internal operational workflow. This workflow involves the inventory app for tracking devices, an onboarding app where customers define their site needs, and the provisioning app, which the office team uses to test and configure devices before shipment. The goal of the provisioning app is to move device provisioning away from the main platform into a seamless application.
    
- Modernizing Primary and Secondary Colors: Michael Weaver reported that they implemented the primary color change and might follow up with a secondary color. Christopher Baron confirmed they want to modernize the application slowly, making it a customer-only space that is modern and fast. Christopher Baron will review Michael Weaver's code this week, stressing that the goal is modernization without breaking existing functionality.
    
- Smoke Detector UUID and Customer Onboarding Updates: Ritwik Rastogi confirmed that the pull request for the UUID switch for smoke detectors is ready. Ritwik Rastogi also completed updates to customer onboarding, making changes so that it no longer redirects, and includes token refreshing and other fixes.
    
- Ongoing Backend Work and S3 Buckets: Ritwik Rastogi is currently working on making S3 buckets private for user home images, site images, and user images. Furthermore, Ritwik Rastogi has a basic backend PR for the provisioning app's basic endpoints ready for review, with other changes queued up once that PR is merged. Christopher Baron acknowledged the need to review the backend code for the provisioning app and clarified that the S3 bucket privacy work is currently underway, not blocked by a review.
    
- Front-end Automated Testing Strategy: Christopher Baron is considering writing automated front-end tests, specifically for caseload functionality, using Playwright. They want Michael Weaver to continue testing manually to understand the product, but they are also open to Michael Weaver writing basic tests, such as creating, editing, and deleting caseloads, with the possibility of another team member writing more comprehensive tests to introduce a different perspective. Michael Weaver agreed that they could write basic tests, and someone else could add tests for edge cases, and suggested adding tests for logging in and logging out.
    
- Testing Tool and Environment Discussion: Christopher Baron expressed a preference for Playwright over Cypress. They discussed creating a dedicated end-to-end testing environment, potentially running a front-end against an actual test backend, perhaps within AWS using something like CodeBuild to interact with cloud resources like WebSockets. Michael Weaver suggested using Datadog, which they find expensive, but Christopher Baron preferred utilizing existing internal logs and resources for analysis, noting that they do not want to use another external provider.
    
- External Testing Tools and Final Remarks: Michael Weaver suggested Valtown as a tool, comparing it to CodePen and describing it as "Twitter for code," that runs on Deno and can be used to build small tests that check if a website is up. Christopher Baron requested a link to Valtown, viewing it as a potential external tool to monitor if the website is operational outside of AWS. Christopher Baron concluded the meeting by offering support and encouraging the team to communicate if they feel blocked or stressed .
